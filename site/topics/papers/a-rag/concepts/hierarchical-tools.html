A-RAG paper

    Think of it like…

    A library with three access modes: card catalog (keywords), subject index (semantic), and reading room (full page). You pick the coarsest mode that still narrows the search.

    Hierarchical Retrieval Tools

    TL;DR: A-RAG exposes keyword_search, semantic_search, and chunk_read — three granularities the agent chooses per step.

    Why this matters

    One-shot top-k treats every question the same — dump ~5K tokens of chunks. Multi-hop QA needs exact entity lookup and semantic bridging. A single retrieval mode cannot serve both.

    Granularity zoom

    Drag from whole chunk → snippet → line → span. This is how A-RAG avoids loading full text until previews justify it.

    Mechanism (§3.2)

    keyword_search — runtime exact match, returns top-k chunk IDs + keyword sentences.
    semantic_search — sentence embeddings, cosine similarity, aggregated to chunks.
    chunk_read — full chunk text (+ adjacent chunks optional) after snippets look promising.
