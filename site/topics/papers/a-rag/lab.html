      Part 1 — Understand

      arXiv:2602.03442 · ~15 min read before playgrounds

      You'll leave Part 1 able to: explain what A-RAG is, why fixed RAG pipelines fail to scale with model capability, and what the paper actually proved.

      Think of it like…

      Two ways to research a question in a library:

      • Naive RAG: A clerk photocopies 5 random pages and hands them to you. You read them once and answer — no going back.
      • A-RAG: You get a card catalog (keyword search), a subject index (semantic search), and a reading room pass (chunk read). You decide what to look up, preview snippets, and only read full pages when needed.

      A-RAG: Scaling Agentic RAG via Hierarchical Retrieval Interfaces

      Agentic RAG should expose hierarchical retrieval interfaces to the model — not fixed one-shot pipelines or human-designed workflows.

      TL;DR: Give the agent three tools — keyword_search, semantic_search, chunk_read — at keyword, sentence, and chunk granularity. The model chooses when and how to retrieve. Beats Graph-RAG and Workflow-RAG on multi-hop QA with comparable or fewer tokens. Scales with model size and test-time compute.

      How to use this page: Read each section in order. Part 2 playgrounds only make sense after Part 1 clicks.

      1. The problem

      Frontier models can reason and use tools over long horizons. But most RAG systems still treat the model as a passive reader — not a retrieval participant.

      Existing RAG falls into two paradigms that exclude the model from retrieval decisions:

      The failure chain

      When the model cannot steer retrieval, capability improvements don't transfer:

      Naive RAG

      Algorithm embeds query, retrieves top-k, dumps chunks. Model never chooses what or when to retrieve.

      Workflow RAG

      Human designer fixes the steps (IRCoT, FLARE, MA-RAG). Model executes a script — cannot adapt strategy per task.

      Scaling blocked

      Stronger reasoning models can't use their capability — retrieval policy is frozen outside the model.

      Takeaway: Even "Naive Agentic RAG" (one embedding tool, model in the loop) already beats fixed pipelines — autonomy alone helps.

      2. The insight — hierarchical interfaces

      Corpus information exists at multiple granularities. A-RAG exposes three tools so the agent picks the right level per step:

      keyword_search

      Lexical · runtime exact match

      1. Agent supplies keyword list
      2. Score chunks by term frequency
      3. Return top-k IDs + matching sentence snippets
      4. Agent decides: read full chunk or search again

      semantic_search + chunk_read

      Sentence embeddings · full chunk access

      1. Encode query → cosine similarity on sentences
      2. Return top-k chunk IDs + best sentences
      3. Agent calls chunk_read for promising IDs
      4. Context tracker dedups re-reads (0 tokens)

      Three principles of agentic autonomy (Figure 2)

      Autonomous strategy
      Model chooses tools and order — not a fixed workflow or external router.
      Iterative execution
      Multi-round retrieval adapts to intermediate observations — not one-shot.
      Interleaved tool use
      ReAct loop: reason → act → observe → reason again. A-RAG is the only compared method satisfying all three.

      You'll verify paradigms and granularity in Part 2.

      Takeaway: Progressive disclosure — snippets first, full chunks on demand — is why A-RAG uses fewer tokens than Naive RAG.

      3. Method — how A-RAG works

      A-RAG has three components: a lightweight hierarchical index, three retrieval tools, and a simple ReAct agent loop.

      • Index — ~1,000-token chunks (sentence-aligned); sentence embeddings via Qwen3-Embedding-0.6B; no offline keyword index
      • Tools — keyword_search, semantic_search, chunk_read
      • Agent loop — one tool per iteration, max L steps; deliberately no parallel calls

      Context tracker

      Maintains set Cread of chunk IDs already loaded. Re-reading returns "This chunk has been read before" — zero additional tokens. Encourages exploring new evidence instead of loops.

      A-RAG variants

      VariantToolsPurpose
      A-RAG (Naive)semantic_search onlyIsolate effect of agentic autonomy alone

      4. Evidence — what the paper proved

      Evaluated on HotpotQA, 2WikiMultiHopQA, MuSiQue, and GraphRAG-Bench. Same LLM backbone across conditions — the paper swaps the retrieval paradigm, not the model.

      BenchmarkNaive RAGA-RAG (Full)Tokens (A-RAG)
      MuSiQue (GPT-5-mini) 52.8% LLM-Acc 74.1% (+21.3) 5,663 vs 5,387 naive
      HotpotQA (GPT-5-mini) 81.2% 94.5% (+13.3) 2,737 vs 5,358 naive (−49%)
      2WikiMultiHop (GPT-5-mini) 50.2% 89.7% (+39.5) 2,930 vs 5,506 naive (−47%)
      GraphRAG-Bench Novel 70.6% 85.3% (+14.7) 6,087 vs 4,997 naive

      Key pattern: A-RAG (Naive) without hierarchy uses more tokens (56K on MuSiQue) with lower accuracy than A-RAG (Full). Hierarchy enables selective reading.

      Test-time scaling: max steps 5→20 gives ~8% on MuSiQue; reasoning effort minimal→high gives ~25%.

      Predict these yourself in Part 2 → Evidence lens.

      Takeaway: Better accuracy with fewer tokens on HotpotQA and 2Wiki — hierarchy beats dump-all retrieval.
