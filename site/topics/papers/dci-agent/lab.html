      Part 1 — Understand

      arXiv:2605.05242 · ~15 min read before playgrounds

      You'll leave Part 1 able to: explain what DCI is, why top-k retrieval fails for agents, and what the paper actually proved.

      Think of it like…

      Two ways to find evidence in a case file:

      • Retriever-mediated: A clerk hands you 5 pre-selected pages. You can't flip to page 47 or grep for an exact SKU.
      • Direct corpus interaction (DCI): You sit at a terminal — grep, pipe, read lines, refine until you hit the exact span.

      Direct Corpus Interaction

      Retrieval for capable agents is an interface-design problem — a terminal gives higher interface resolution than a fixed top-k API.

      TL;DR: Skip the vector index. Let the agent grep, read, and pipe commands on raw files — like a researcher at a terminal. Strong results on agentic search, multi-hop QA, and IR benchmarks.

      How to use this page: Read each section in order. Part 2 playgrounds only make sense after Part 1 clicks.

      1. The problem

      Baseline RAG works for single-shot Q&A: embed query, retrieve top-k chunks, generate answer. But agentic search is different — the agent runs many steps, combines weak clues, and verifies hypotheses against local context.

      A fixed top-k retriever API decides too early what the agent is allowed to see. Evidence filtered at this step cannot be recovered — no second chance to grep page 47.

      The failure chain

      Without a composable interface, agentic search breaks down:

      Top-k filter

      Retriever ranks chunks once. Gold evidence at rank #12 never reaches the agent.

      Lost evidence

      Agent can't pipe "grep warranty | grep valve" or re-query with a refined pattern.

      Agent guesses

      Without exact spans, the model hallucinates or cites coarse, unverifiable snippets.

      Takeaway: Top-k is a one-way door — evidence dropped early never comes back.

      2. The insight — two paradigms

      The paper's core reframing: retrieval is not just retriever design — it's interface design. Same agent goal, two ways to access the corpus:

      Retriever-mediated

      Offline index · top-k API

      1. Embed query → single vector
      2. Search ANN index over chunks
      3. Return top-k ranked snippets
      4. Agent reads — cannot re-query differently

      Direct corpus interaction

      Raw files · grep, read, pipe

      1. Agent plans tool sequence
      2. grep / rg on raw corpus
      3. Pipe & refine weak clues
      4. Read exact line → verify & cite

      Two terms to remember

      Interface resolution
      Granularity of corpus access: whole document → passage → line → exact span. DCI operates at finer units than chunk retrieval.
      Coverage vs localization
      Coverage: did the trajectory surface gold documents at all? Localization: how precise is the exposed snippet? DCI often ties on coverage but wins on localization.

      You'll verify both paradigms in Part 2.

      Takeaway: Same agent, same corpus — the interface (top-k API vs terminal) changes what evidence is reachable.

      3. Method — how DCI works

      DCI gives the agent direct access to a raw corpus via terminal tools — no embeddings, no vector index:

      • Raw corpus — files on disk, not pre-chunked embeddings
      • Terminal tools — grep, rg, find, read, shell pipes
      • Agent loop — plan → search → refine → verify across many steps

      Two implementations

      DCI-Agent-Lite — Pi-based harness, bash + read only, lightweight context management for reproducibility.

      DCI-Agent-CC — Claude Code harness, same DCI interface, used for main benchmark numbers.

      Context management (L0–L4)

      4. Evidence — what the paper proved

      Same LLM backbone across conditions — the paper swaps the retrieval interface, not the model. Key numbers:

      BenchmarkBaselineScoreDCI-Agent-CCScore
      BrowseComp-Plus Retriever (Qwen3-Embed-8B) 69.0% · $1,440 DCI-Agent-CC 80.0% · $1,016 (−29% cost)
      Multi-hop QA (avg) ASearcher-Local-14B 52.3% DCI-Agent-CC 83.0% (+30.7)
      IR ranking (avg NDCG@10) ReasonRank-32B 47.0% DCI-Agent-CC 68.5% (+21.5)

      The takeaway: composable grep-based search beats strong sparse/dense/rerank baselines on agentic tasks — especially where exact localization matters.

      Predict these yourself in Part 2 → Evidence lens.

      Takeaway: Same LLM backbone — swapping the interface (not the model) drove +11 pts accuracy and −29% cost on BrowseComp.
