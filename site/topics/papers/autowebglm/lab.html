      Part 1 — Understand

      arXiv:2404.03648 · KDD 2024 · ~15 min read before playgrounds

      You'll leave Part 1 able to: explain why raw HTML breaks LLM web agents, what AutoWebGLM does differently, and what the benchmarks actually show.

      Think of it like…

      Two ways to shop online with a helper:

      • GPT-4 + raw HTML: Your helper gets the entire page source — ads, tracking scripts, 50,000 tokens — and plays 20 questions to guess which button to click.
      • AutoWebGLM: A clerk highlights only the clickable parts, remembers what you already tried, and says "click button #47" in one shot — trained on thousands of real browsing traces.

      AutoWebGLM: A Large Language Model-based Web Navigating Agent

      Web navigation is an observation-design problem — prune HTML before the model sees it, then bootstrap a 6B agent with curriculum data and rejection sampling.

      TL;DR: ChatGLM3-6B agent with HTML Pruner + unified actions. Hybrid human–AI training data (~10k traces), DPO alignment, RFT in MiniWoB++/WebArena. Beats GPT-4 on AutoWebBench (64.8% vs 38.6% SSR) and MiniWoB++ (89.3% vs 32.1%). Introduces bilingual AutoWebBench.

      How to use this page: Read each section in order. Part 2 playgrounds only make sense after Part 1 clicks.

      1. The problem

      LLM web agents exist, but most fail on real sites. The paper names three root causes:

      Complex HTML

      Pages are huge DOM trees — scripts, styles, ads. Raw HTML overflows context and hides the one button that matters.

      Versatile actions

      Click, type, scroll, tabs, dropdowns — inconsistent schemas across agents make generalization hard.

      Open-domain web

      Infinite site layouts and languages — imitation on a few benchmarks does not transfer to unseen domains.

      Prior work like MindAct filters elements with many LLM rounds; WebAgent uses 540B Html-T5. AutoWebGLM asks: can a deployable 6B model win if the interface and training match human browsing?

      Takeaway: The bottleneck is often what the model sees, not just model size.

      2. The insight — simplify observations, unify actions

      Inspired by human browsing: people ignore boilerplate and focus on interactive elements. AutoWebGLM mirrors that in software.

      HTML Pruner

      Algorithm 1 · keep local tree context

      1. Tag operable elements (clickable, typeable)
      2. Keep ancestors, descendants, siblings up to depth limits
      3. Drop empty decorative nodes
      4. Feed ~4K-token simplified tree to the policy

      Unified observation + action

      Single policy call per step

      1. Task description + simplified HTML
      2. Scroll position + page height
      3. Explicit previous-action history
      4. Output: click(id), type_string, finish, …

      You'll verify the observation zoom in Part 2.

      Takeaway: Compression + memory — prune noise, remember past clicks — beats dumping raw HTML into GPT-4.

      3. Method — data, curriculum, bootstrap

      Building a web agent needs data LLMs cannot generate alone. AutoWebGLM uses hybrid human–AI construction and multi-stage training.

      Hybrid data construction (§4.1)

      • Stage 1 — webpage recognition + simple operations (Evol-Instruct style task generation, human filter)
      • Stage 2 — complex multi-step tasks; humans record traces via browser plugin; LLM fills Chain-of-Thought rationales
      • Merged with Mind2Web + MiniWoB++ training splits (~10k complex traces for AutoWebBench eval)

      Training stages (§4.2)

      StageTechniqueWhat it fixes
      SFTCurriculum imitationBasic DOM comprehension + action format
      DPOPrefer successful trajectoriesWrong element clicks, incoherent plans
      RFTSample 64× in sandbox, keep successesMiniWoB++ / WebArena specialization

      Explore observation levels in Part 2 → Observation space and training stages in Training curriculum.

      Takeaway: Complex human–AI traces (+Stage 2) give the biggest Mind2Web jump; RFT is what pushes MiniWoB++ to 89.3%.

      4. Evidence — what the paper proved

      Evaluated on AutoWebBench (new), Mind2Web, MiniWoB++, and WebArena. Metric: Step Success Rate (SSR) or task success rate.

      BenchmarkGPT-4AutoWebGLM 6BNotes
      AutoWebBench EN (cross-task SSR) 38.6% 64.8% Real bilingual site traces
      Mind2Web (avg SSR) 30.9% 59.5% Offline real-website tasks
      MiniWoB++ (task success) 32.1% 89.3% After RFT on 56 simulated tasks
      WebArena (task success) 14.4% 18.2% Live sandbox — still hard

      Ablation: train-set-only Mind2Web 48.1% → +Stage 2 56.7% → +DPO 59.5%. RFT adds MiniWoB++/WebArena gains without improving Mind2Web further.

      Failure analysis: hallucinations 44%, poor graphical recognition 28%, task misinterpretation 20%, pop-ups 8%.

      Predict winners in Part 2 → Evidence lens.

      Takeaway: Strong on curated benchmarks; WebArena 18.2% is the honest ceiling — production browsing is not solved.
