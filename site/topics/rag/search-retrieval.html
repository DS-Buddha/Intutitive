RAG pipeline

    Think of it like...

    Dense search is like finding books by topic ("something about warranties"). BM25 is like finding books that contain the exact product code. Hybrid search uses both catalogs.

    Search & Retrieval

    TL;DR: Search maps a query to candidate evidence before generation. Dense vs sparse vs hybrid — what gets into context usually dominates RAG quality more than the LLM prompt.

    Why this matters

    Search maps a query to candidate evidence before generation. Dense vs sparse vs hybrid — what gets into context usually dominates RAG quality more than the LLM prompt.

    Without this: downstream retrieval and generation inherit upstream mistakes — bigger models and rerankers rarely recover.
    With this: each pipeline stage receives clean, structured input — grounding becomes possible instead of hopeful.

    Play with the machinery

    Same query, three ranking strategies side-by-side. Edit the query to see when each wins.

    How it works

    Two-stage retrieval is common: cheap recall (bi-encoder, BM25, hybrid) then precision (reranker, late interaction).

    • BM25: exact term matching, great for SKUs and IDs
    • Dense: semantic similarity, great for paraphrases
    • RRF fusion: combine rank lists without score calibration

    Tradeoffs

    Use when

    • Mixed query types — exact IDs and natural language
    • When dense-only misses keyword-critical passages

    Skip when

    • Homogeneous short notes where dense-only suffices
