RAG pipeline

    Think of it like...

    You ask a librarian a question. Instead of going to the library, searching shelves, and reading every book, the librarian:
    1. Breaks your question into search terms (chunking)
    2. Looks them up in the card catalog (embedding)
    3. Pulls the 3 most relevant books (retrieval)
    4. Reads from those books to answer you (generation)
    Without this process, the librarian would just make something up. Baseline RAG is that process for LLMs.

    Baseline RAG

    TL;DR: The simplest end-to-end pipeline: split documents into chunks, embed them, retrieve relevant chunks based on a query, feed them to an LLM to generate grounded answers.

    Why this matters

    Large language models are powerful at language, but they have a critical limitation: they only know what they learned during training. They don't have access to your documents, databases, or real-time information. When you ask them a question outside their training data, they confidently make things up — a phenomenon called hallucination.

    Baseline RAG solves this by giving the LLM access to your documents at generation time. Instead of answering from memory, the model answers from sources. This is the foundation of every RAG system.

    The failure chain

    Without each step, the entire system breaks down:

    No chunking

    Embed 100-page PDFs as single vectors → slow, loses detail

    No embedding

    Can't measure similarity → can't retrieve relevant chunks

    No retrieval

    LLM has no sources → guesses, hallucinates

    No generation

    You have sources but can't synthesize answers from them

    Without RAG:

    User: "What is our Q3 revenue?"
    LLM (hallucinating): "Your Q3 2023 revenue was $5.2 billion."
    Reality: You have no idea if that's true.

    With baseline RAG:

    User: "What is our Q3 revenue?"
    LLM (from your document): "According to your Q3 earnings report, revenue was $4.8 billion."
    Reality: You can verify the source.

    Quick overview — 4-step pipeline walkthrough

    How it works

    Now that you understand the why, let's look at the mechanism. The baseline pipeline is beautifully simple: it's just four sequential operations.

    The four-step baseline

    1

    Chunking

    Split documents into fixed-size pieces (e.g., 512 tokens). This is usually done with a sliding window or by splitting on sentence/paragraph boundaries. Simple but effective.

    2

    Embedding

    Pass each chunk through an embedding model (e.g., text-embedding-3-small). Each chunk becomes a 384–1536 dimensional vector. Store these vectors in a vector database.

    3

    Retrieval

    When a user asks a question, embed the query and find the top-k chunks with highest cosine similarity. This is a nearest-neighbor search in vector space.

    4

    Generation

    Pass the retrieved chunks (context) + user query to an LLM. The model generates an answer based on the grounded context instead of pure training data.

    Technical details

    Chunking parameters

    • Chunk size: 256–1024 tokens typical. Larger chunks preserve context, smaller chunks increase retrieval precision.
    • Overlap: 20–50% overlap prevents information from being split awkwardly across chunks.
    • Strategy: Fixed size, sentence-aware, or semantic chunking (coming in Document Ingestion lesson).

    Embedding model choice

    • Dense embeddings: Capture semantic meaning. Models: OpenAI text-embedding-3, Cohere, BGE.
    • Dimension: 384–1536 dims typical. More dims = more expressive but slower.
    • Normalized: Most modern models output unit-normalized vectors (dot product = cosine similarity).

    Retrieval: cosine similarity

    For each query, compute: similarity = (query_vector · chunk_vector) / (||query|| · ||chunk||)

    Return top-k chunks by similarity score (usually k=3–5).

    Generation: prompt engineering

    The prompt typically looks like:

    Given the following context:
<RETRIEVED_CHUNKS>

Answer this question: <USER_QUERY>

    The LLM must generate an answer that depends on the context. If context is irrelevant, the answer will be hallucinated or off-topic.

    Pseudocode

    # Indexing phase (one-time, offline)
documents = load_documents()
chunks = []
for doc in documents:
    chunks.extend(split_into_chunks(doc, size=512, overlap=0.2))

embeddings = embed_model.encode(chunks)  # [N, 384]
vector_db.insert(chunks, embeddings)

# Query phase (at runtime)
user_query = "What is baseline RAG?"
query_embedding = embed_model.encode(user_query)  # [384]

# Retrieve
scores = cosine_similarity(query_embedding, embeddings)  # [N]
top_indices = argsort(scores)[-5:]  # top 5
retrieved_chunks = [chunks[i] for i in top_indices]

# Generate
context = "\n\n".join(retrieved_chunks)
prompt = f"""Given the context:
{context}

Question: {user_query}
Answer:"""

answer = llm.generate(prompt)
print(answer)

    This is pseudocode. Real implementations add error handling, batching, caching, reranking, and much more. But the core is this simple.

    Tradeoffs

    Use baseline RAG when

    • You have a document corpus and want Q&A over it
    • You need explainability (retrieve sources alongside answers)
    • You want to avoid hallucination on domain-specific questions
    • You're prototyping and need something that works immediately
    • Your corpus is small-to-medium (< 1M documents)

    Poor fit when

    • Questions require reasoning across multiple documents (no aggregation)
    • You need to find low-frequency information (top-5 retrieval may miss it)
    • Your corpus is massive (billions of documents) — retrieval becomes slow
    • Chunks lose context when isolated (needs hierarchical indexing instead)
    • You need to preserve document structure (PDF tables, etc.)

    Cost/Benefit Analysis

    Dimension Cost Benefit
    Latency ~500ms (embed query + vector search + LLM) Acceptable for most applications
    Index size 1GB per million chunks (10M for 1M docs) Manageable on commodity hardware
    Accuracy No guarantee relevant info is top-5 Dramatically reduces hallucination vs. zero-shot
    Setup time 30 mins to 2 hours (chunk, embed, index) One-time cost, then live immediately
    Cost per query Embedding + LLM calls (~0.1–1 cent) Cheaper than hiring a researcher

    Next: Why Baseline Isn't Enough

    Baseline RAG works, but it has blind spots. What if your top-5 chunks don't contain the answer? What if your document is 500 pages and the answer spans pages 10 and 400 (no single chunk contains both)? What if your embedding model is poor at understanding domain jargon?

    How it connects

    Baseline RAG is the foundation. Everything else is an improvement to one of its four steps. Here's where this concept fits in the RAG pipeline:

    → Document Ingestion (Next)

    Improves chunking with layout-aware parsing. Instead of naive splitting, understand document structure (tables, headings, etc.).

    → Search & Retrieval

    Improves embedding and retrieval. Use better embedding models, hybrid search (dense + sparse), reranking.

    → Indexing & Structure

    Improves retrieval via hierarchical indexing (RAPTOR). Instead of flat chunks, build a tree of summaries.

    → Verification & Agentic Loops

    Improves generation quality. Have the LLM assess whether it has enough information, retrieve more if needed (self-reflection).

    No prerequisites: Baseline RAG assumes you understand what embeddings are and why similarity search works. These are standard ML concepts (not covered here).
